Modern AI applications are not single-shot requests. They involve multi-turn conversations, tool calls, branching reasoning paths, and long-running workflows that must survive process restarts. This paper describes the execution model Arkloop uses to represent these as first-class *threads*.

## Background

Early LLM applications modeled conversations as stateless request-response cycles: send a context window, receive a completion, repeat. This works well at small scale but breaks down under three conditions:

- **Duration**: Reasoning chains that exceed a single response window
- **Multiplicity**: Concurrent agent invocations that share state
- **Durability**: Workflows that must resume after failure or restart

The thread model addresses all three.

## The Thread Abstraction

A thread in Arkloop is an ordered, append-only log of *events*. Each event has a type (`user_message`, `assistant_message`, `tool_call`, `tool_result`), a stable identifier, and an immutable payload. The log is the source of truth—no derived state is stored outside it.

```go
type Thread struct {
    ID        string
    CreatedAt time.Time
    Events    []Event
    Status    ThreadStatus
}

type Event struct {
    ID        string
    ThreadID  string
    Type      EventType
    Payload   json.RawMessage
    CreatedAt time.Time
}
```

This structure has a useful property: any snapshot of the thread at time *t* can be reconstructed by replaying events up to *t*. There is no "current state" that can become inconsistent.

## Run Lifecycle

Executing a thread produces a *run*—a bounded, tracked invocation that advances the thread by one or more events. Runs are separate entities from threads:

```
Thread  ──────────────────────────────────────▶  (append-only)
           Run₁       Run₂       Run₃
           [start──end] [start──end] [start──...]
```

A run transitions through states: `pending → running → succeeded | failed | cancelled`. The terminal states are permanent. If a run fails mid-execution, the thread log up to the last committed event remains intact, and a new run can resume from there.

## Concurrency Model

At any point, at most one run can be in the `running` state for a given thread. This is enforced at the storage layer with a row-level lock, not at the application layer. Application code does not need to reason about mutual exclusion.

Concurrent *reads* are unrestricted. Multiple consumers can stream events from the same thread simultaneously—useful for multiplexing a single agent execution to several connected clients.

## Tradeoffs

The append-only log is not free. Replaying a long thread to reconstruct context is O(n) in the number of events. For threads with thousands of events, this matters.

Arkloop mitigates this with *checkpoints*: periodic snapshots of the constructed context window that allow replay to start from a recent point rather than the beginning. Checkpoints are written asynchronously and are advisory—the log remains the canonical source.

A second tradeoff is storage amplification. Tool results and large assistant responses are stored verbatim in the event log. Arkloop applies content-addressed deduplication for binary attachments, but text content is stored once per occurrence.

## Conclusion

The thread execution model trades some storage efficiency for strong consistency and resumability guarantees. For AI agent workflows where correctness and debuggability matter more than throughput, these are the right tradeoffs. The model also provides a natural audit log: every action taken by an agent is recorded as an event, with a timestamp and a causal chain back to the initiating user message.
